1.2 MVC Sederhana

📗 Minggu 1 ⏱️ 25 menit 📖 Teori

📌 Prasyarat

  • Sudah membaca materi 1.1 Pengenalan Laravel
  • Paham konsep dasar: Laravel = framework PHP dengan fitur lengkap

🎯 Tujuan Belajar

Setelah menyelesaikan materi ini, kamu akan mampu:

  • Menjelaskan apa itu MVC dan mengapa pola ini penting
  • Mengidentifikasi peran Model, View, dan Controller masing-masing
  • Menelusuri alur request dari user sampai tampil di browser
  • Menunjukkan lokasi file Model, View, dan Controller di folder Laravel

💡 Apa itu MVC?

MVC singkatan dari Model - View - Controller. Ini adalah cara mengorganisir kode supaya rapi, mudah dibaca, dan mudah dikembangkan.

🍽️

Analogi: Restoran

Bayangkan kamu pergi ke restoran. Ada 3 bagian yang bekerja sama:

  • 🧑‍🍳 Dapur (Model) — Tempat memasak makanan dan menyimpan resep. Tidak berhubungan langsung dengan pelanggan.
  • 🍽️ Meja & Piring (View) — Penyajian makanan di meja. Yang dilihat pelanggan.
  • 🧑‍💼 Pelayan (Controller) — Menerima pesanan pelanggan, minta dapur masak, lalu bawa makanan ke meja.

Peran mereka terpisah tapi saling bekerja sama. Kalau koki merangkap pelayan, akan kacau! Begitu juga di kode.

🔍 Peran Masing-Masing Komponen

📊

Model

Tugas: Mengelola DATA — ambil, simpan, update, hapus data dari database.

Contoh di kehidupan:

  • Gudang & resep di restoran
  • Database siswa di sekolah
  • Lemari arsip di kantor

📁 Lokasi: app/Models/

👁️

View

Tugas: Menampilkan TAMPILAN — halaman HTML yang dilihat user di browser.

Contoh di kehidupan:

  • Meja & piring di restoran
  • Rapor yang dicetak
  • Layar ATM

📁 Lokasi: resources/views/

🎮

Controller

Tugas: MENGHUBUNGKAN Model & View — terima request, proses, kirim response.

Contoh di kehidupan:

  • Pelayan di restoran
  • Teller di bank
  • Kasir di toko

📁 Lokasi: app/Http/Controllers/

🔄 Alur Request di Laravel (Step by Step)

Ketika user mengakses sebuah halaman, ini yang terjadi di balik layar:

👤 User
klik/ketik URL
→
🛣️ Route
terima request
→
🎮 Controller
proses logika
→
📊 Model
ambil data
→
👁️ View
tampilkan
→
👤 User
lihat halaman

Mari kita ikuti alurnya dengan contoh nyata: user ingin melihat daftar buku.

1

🛣️ Route — Penerima Tamu

User mengetik sekolahku.com/buku. Route mencocokkan URL ini dengan Controller yang tepat.

routes/web.php 
// Kalau ada yang buka URL '/buku', jalankan method index di BookController
Route::get('/buku', [BookController::class, 'index']);
🔍

Penjelasan Baris per Baris

  • Route::get() — Tangkap request GET (ketik URL atau klik link)
  • '/buku' — URL yang ditangkap, yaitu /buku
  • [BookController::class, 'index'] — Kirim ke method index di BookController
2

🎮 Controller — Si Pelayan

Controller menerima request, minta data ke Model, lalu kirim hasilnya ke View.

app/Http/Controllers/BookController.php 
namespace App\Http\Controllers;

class BookController extends Controller
{
    public function index()
    {
        // Langkah 1: Minta semua data buku ke Model
        $books = Book::all();

        // Langkah 2: Kirim data ke View untuk ditampilkan
        return view('books.index', ['books' => $books]);
    }
}
🔍

Penjelasan Baris per Baris

  • Book::all() — Panggil Model Book, ambil semua data buku dari database
  • $books — Variabel yang menampung semua data buku
  • view('books.index', [...]) — Kirim ke file View resources/views/books/index.blade.php
  • ['books' => $books] — Kirim data $books ke View dengan nama variabel $books
3

📊 Model — Gudang Data

Model berinteraksi dengan database. Cukup definisikan nama tabel, Laravel yang urus sisanya.

app/Models/Book.php 
namespace App\Models;

use Illuminate\Database\Eloquent\Model;

class Book extends Model
{
    // Laravel otomatis tahu tabel = 'books' (nama class + s)
    // Kolom: id, title, author, created_at, updated_at
}
🤯

Keren, bukan?

Model Book tidak perlu menulis SQL apapun! Laravel otomatis tahu untuk mengakses tabel books (huruf kecil + tambah 's'). Ini disebut Convention over Configuration — ikuti aturan penamaan, semuanya otomatis bekerja.

4

👁️ View — Tampilan Cantik

View menampilkan data yang dikirim Controller dalam bentuk HTML yang bisa dilihat user.

resources/views/books/index.blade.php 
<h1>📚 Daftar Buku</h1>

<table>
    <tr>
        <th>Judul</th>
        <th>Penulis</th>
    </tr>
    @foreach($books as $book)
    <tr>
        <td>{{ $book->title }}</td>
        <td>{{ $book->author }}</td>
    </tr>
    @endforeach
</table>
🔍

Penjelasan Baris per Baris

  • @foreach($books as $book) — Loop setiap buku. Seperti foreach PHP biasa tapi sintaks Blade
  • {{ $book->title }} — Tampilkan judul buku. Tanda {{ }} akan otomatis escape HTML (aman dari XSS)
  • @endforeach — Akhiri loop

📁 Dimana File-file Ini Berada?

Berikut struktur folder Laravel yang perlu kamu tahu:

📂 Struktur Folder Laravel (Simplified) 
📦 project-laravel/
├── 📂 app/
│   ├── 📂 Http/Controllers/    ← Controller di sini
│   │   └── BookController.php
│   └── 📂 Models/              ← Model di sini
│       └── Book.php
├── 📂 resources/views/        ← View di sini
│   └── 📂 books/
│       └── index.blade.php
├── 📂 routes/                  ← Route di sini
│   └── web.php
├── 📂 database/                 ← Migration & seeder
├── 📂 public/                   ← CSS, JS, gambar
└── 📂 config/                   ← Pengaturan app
💡

Tips Mengingat Lokasi File

Model → app/Models/
View → resources/views/
Controller → app/Http/Controllers/
Route → routes/web.php (satu-satunya yang berbeda pola)

❓ Kenapa Harus Dipisah? Tidak Ribet?

Pertanyaan bagus! Ini jawabannya:

Aspek ❌ Tanpa MVC (Campur Aduk) ✅ Dengan MVC (Terpisah)
Baca kode Sulit — HTML, PHP, SQL bercampur di satu file Mudah — setiap file punya tugas jelas
Cari bug Bingung cari di mana errornya Langsung tahu: error data? cek Model. Error tampilan? cek View.
Kerja tim Saling tumpuk — satu orang edit, yang lain kacau Bisa bagi tugas: A bikin View, B bikin Controller
Ubah desain Harus edit banyak file karena campur dengan logika Edit View saja, logika dan data tetap aman

⚠️ Kesalahan Umum Pemula

❌ Menulis query SQL langsung di View

Contoh: <?php $data = mysqli_query($conn, "SELECT * FROM buku"); ?> langsung di file HTML.

✅ Query harus di Model, bukan di View. View hanya untuk MENAMPILKAN data yang sudah dikirim dari Controller.

❌ Controller yang terlalu "gemuk"

Memasukkan semua logika bisnis (validasi, hitung diskon, kirim email) ke dalam satu method Controller.

✅ Controller hanya jadi "penghubung". Logika berat taruh di Model atau Service class.

❌ Bingung nama file Model vs nama tabel

Model bernama Book tapi tabel database bernama buku — tidak cocok!

✅ Ikuti konvensi Laravel: Model Book (singular, PascalCase) → tabel books (plural, snake_case).

📋 Ringkasan Materi

  • MVC = pola untuk memisahkan kode menjadi 3 bagian: Model, View, Controller
  • Model → mengelola data dan database (📁 app/Models/)
  • View → menampilkan halaman ke user (📁 resources/views/)
  • Controller → menghubungkan Model dan View (📁 app/Http/Controllers/)
  • Route → mengatur URL mana masuk ke Controller mana (📁 routes/web.php)
  • Alur: User → Route → Controller → Model → View → User

📝 Latihan

🟢 Level 1 — Dasar

  1. Gambar diagram alur MVC dengan kata-kata dan contohmu sendiri (boleh pakai kertas).
  2. Sebutkan lokasi folder untuk file Model, View, Controller, dan Route di Laravel.
  3. Di file mana kamu harus menulis kode untuk: (a) query database, (b) tampilan HTML, (c) hubungkan keduanya?

🔵 Level 2 — Analisis

  1. Baca kode Controller di atas. Kalau kamu ingin menambah fitur "cari buku berdasarkan judul", method mana yang perlu diubah dan apa yang ditambahkan?
  2. Buat sketsa (gambar tangan) bagaimana alur MVC bekerja untuk fitur "tambah buku baru".

🟣 Level 3 — Berpikir Kritis

  1. Kalau kita TIDAK pakai MVC dan semua kode ditulis di satu file PHP, apa yang terjadi kalau aplikasinya punya 50+ halaman? Jelaskan masalah yang mungkin muncul.
  2. Selain restoran, cari analogi lain untuk MVC dari kehidupan sehari-hari. Jelaskan peran setiap komponen.
